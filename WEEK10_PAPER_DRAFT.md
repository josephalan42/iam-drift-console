# Week 10 Paper Draft (Version 1)

## BLUF
This project demonstrates how AI can improve IAM drift monitoring by turning raw AWS permission changes into readable, risk-focused security guidance. The main value of the system is that it not only detects changes in users, groups, and policies, but also explains why those changes matter, what risk they introduce, and what actions should be taken next. The outcome is a more practical security workflow that helps users move from detection to response faster while also highlighting areas that still need hardening, such as dependency risk, secure deployment settings, and continued validation of AI output.

## Background
Cloud IAM environments change frequently, and even small policy updates can create major security exposure if they are not reviewed quickly. In many organizations, administrators can see that something changed, but they still have to spend additional time interpreting what the change means, which services are affected, and whether the permission set creates escalation risk. That delay creates both operational and security problems.

This project addresses that gap by combining IAM drift detection with AI-assisted explanation. The system compares a baseline snapshot against a current snapshot, identifies drift in users and policy assignments, and then generates recommendations that translate technical IAM changes into security-focused summaries. A dashboard presents the results in a more readable format so that users can review alerts, filter findings, and examine recommendations without reading raw JSON output. Strategically, this matters because it connects security monitoring with decision support rather than stopping at detection alone.

## Methodology
The project was built around three main components: drift detection, AI recommendation generation, and dashboard presentation.

First, the drift detection logic compares a baseline IAM state against a current snapshot and identifies changes such as added users, inline policy additions, managed policy changes, group membership changes, and resolved managed policy modifications. The design also normalizes policy documents so that ordering differences do not create false positives.

Second, the AI recommendation component improves the usefulness of the alert output. Prompting was tightened so the model behaves more like a focused IAM analyst instead of a general assistant. The prompt now instructs the model to use only the provided fields, stay concise, remain policy-specific and service-specific, and return valid JSON only. The request payload also supplies structured context such as risk flags, extracted policy entries, service context, and an output contract that explicitly defines severity, summary, impact, mitigations, rollback steps, and verification steps. In addition, service-specific recommendation templates were added for IAM, S3, Lambda, EC2, RDS, KMS, and Secrets Manager so that output is more tailored and less generic.

Third, the dashboard was updated to improve usability. Quality-of-life changes included a searchable filter bar, collapsible recommendation sections, clearer layout separation for large blocks of text, separate counts for user events and policy changes, and a more polished dark and light mode experience. These updates were important because the earlier interface made it harder to scan large outputs quickly.

To evaluate the system, testing was planned and partially executed across both functional and security categories. Functional checks focused on whether the drift detector correctly identified policy and user changes and whether the application generated recommendation files and API responses as expected. Security checks included `bandit` for code scanning, `pip-audit` for dependency review, and adversarial test planning for malformed JSON and prompt injection style inputs. AI validation focused on grounding, correctness, severity alignment, and service-specific usefulness.

## Results
Initial testing showed that core functionality worked as intended. The drift detector successfully generated a drift report, and the AI explanation script generated recommendation output tied to those findings. Automated smoke tests using `pytest` passed, showing that key logic and the API path remained functional during iteration.

The prompt and recommendation updates improved output quality in several ways. Recommendations became more specific, more structured, and more relevant to the AWS service involved. Instead of repeated generic advice, the system now produces targeted guidance such as limiting S3 bucket scope, restricting Lambda permissions to required functions, or constraining IAM wildcard actions. This change increased the practical value of the AI output because the recommendations were easier to connect to the actual security problem.

The testing process also revealed meaningful security findings. Static analysis identified that Flask was running with `debug=True`, which should not be used in a production context. Dependency scanning also identified a Flask vulnerability that should be remediated through version upgrade. These findings are valuable because they show that the testing process is not just verifying success cases; it is also exposing areas where the project itself still needs improvement. That strengthens the credibility of the work.

## Strategic Insight
The broader value of this project is not only technical accuracy but decision advantage. Security teams often collect large volumes of alerts, but the bottleneck is understanding which events matter and how to respond. By combining drift detection with structured AI explanation, the project reduces that interpretation gap. In a real environment, that can improve response time, support least-privilege enforcement, and reduce the chance that risky IAM changes go unreviewed.

There is also a strategic lesson in the limitations that were discovered. AI recommendations must be validated, not simply accepted. Security scanning tools may be limited by platform constraints, and user interface design strongly affects whether findings are actually usable under pressure. For that reason, the project shows that successful AI-enabled cybersecurity tools require three things working together: reliable detection, trustworthy recommendation logic, and a usable interface. Focusing on only one of those areas would make the system less effective.

## Conclusion
This draft shows that the project is progressing toward a practical IAM drift monitoring system that combines detection, explanation, and usability. The current version already demonstrates meaningful functionality and clear project value, while the testing results identify specific next steps for improvement. Moving forward, the strongest priorities are to continue refining the AI validation process, resolve the security findings uncovered during testing, and further improve the dashboard for operational use. Overall, the project supports the idea that AI can strengthen cybersecurity workflows when it is paired with structured prompts, validation logic, and security-focused testing.
